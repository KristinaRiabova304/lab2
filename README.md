#  GameQuest API — Асинхронна взаємодія (RabbitMQ)

---

##  Мета роботи

Метою роботи є реалізація асинхронної взаємодії між компонентами системи з використанням message broker.  
У рамках роботи впроваджено event-driven підхід, шаблон Transactional Outbox, механізми обробки помилок, retry, Dead Letter Queue та базове розподілене трасування.

---

##  Завдання роботи

У межах лабораторної роботи було виконано:

- обрано message broker — **RabbitMQ**;
- реалізовано відправку подій після створення та модифікації сутностей;
- створено окремий сервіс-консьюмер для обробки повідомлень;
- реалізовано шаблон **Transactional Outbox**;
- додано симуляцію випадкових помилок обробки повідомлень;
- реалізовано механізм повторних спроб (**retry**);
- налаштовано **Dead Letter Queue (DLQ)**;
- додано базове розподілене трасування логів через **traceId**.

---

##  Оформлення результатів роботи

У результаті виконання роботи створено систему, яка:

- підтримує CRUD-операції над іграми та квестами;
- генерує події після змін даних;
- передає події через RabbitMQ;
- обробляє повідомлення асинхронно;
- виконує retry у випадку помилок;
- перенаправляє невдалі повідомлення в DLQ;
- дозволяє відслідковувати події через traceId.

## 💻 Опис коду програми

Проєкт реалізовано на **Node.js + Express**.

### Основні модулі:

- `server.js` — запуск сервера та ініціалізація брокера  
- `app.js` — REST API  
- `db.js` — in-memory база даних  
- `broker.js` — налаштування RabbitMQ  
- `outbox-publisher.js` — відправка подій з outbox  
- `consumer.js` — обробка повідомлень  
- `dlq-listener.js` — обробка повідомлень з DLQ  
- `tracing.js` — реалізація traceId  

---

## 🔄 Принцип роботи системи

1. Клієнт надсилає HTTP-запит до API  
2. API змінює стан сутності  
3. Подія записується в Outbox  
4. Publisher відправляє подію в RabbitMQ  
5. Consumer отримує повідомлення  

Далі:

- якщо SUCCESS → повідомлення підтверджується (ACK)  
- якщо ERROR → виконується retry  
- якщо перевищено кількість спроб → повідомлення потрапляє в DLQ  
- DLQ Listener логує повідомлення

##  Висновки

У ході роботи було реалізовано асинхронну обробку подій з використанням RabbitMQ.
Було впроваджено шаблон Transactional Outbox, що забезпечує більш надійну доставку повідомлень.

Також реалізовано:

- механізм retry
- Dead Letter Queue
- обробку помилок
- базове трасування

У проєкті використано event-driven підхід, але не повний event sourcing, оскільки джерелом істини залишається поточний стан сутностей.
